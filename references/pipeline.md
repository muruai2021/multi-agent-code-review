# 协作流程

## 整体流程图

```
User Request
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 1: 任务分析                                            │
│  - 识别语言和框架                                             │
│  - 确定审查范围                                               │
│  - 制定审查计划                                               │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 2: 并行审查 (所有 Agent 同时工作)                       │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │
│  │Architecture│ │Security │ │Performance│ │ Quality │ │Testing │ │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘ │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  Phase 3: 报告汇总                                            │
│  - 收集各 Agent 报告                                          │
│  - 去重和优先级排序                                           │

                            │
                            ▼

                            │
                            ▼
                      Final Report
```

## 详细阶段说明

### Phase 1: 任务分析

**主编操作**:
1. 解析用户请求，提取关键信息：
   - 需要审查的文件列表
   - 代码语言和框架
   - 是否有特定的审查重点
2. 确定需要调度的 Agent 数量和类型
3. 制定审查计划

**输出**: 审查计划文档

### Phase 2: 并行审查

**执行策略**: 所有专业 Agent 同时开始工作

**Agent 调度矩阵**:

| 代码语言 | 必须调度的 Agent |
|----------|-----------------|
| TypeScript/JS | Architecture, Security, Performance, Quality, Testing |
| Python | Architecture, Security, Performance, Quality, Testing |
| Go | Architecture, Security, Performance, Quality, Testing |
| 全部语言 | +API Reviewer（如果是 API 项目）|

**并发控制**:
- 最大并发 Agent 数: 5
- 单个 Agent 超时: 5 分钟
- 全部 Agent 超时: 10 分钟

### Phase 3: 报告汇总

**主编操作**:
1. 收集所有 Agent 的报告
2. 合并重复发现的问题
3. 按严重性排序
4. 生成汇总统计

**合并规则**:
- 同一文件的同一行只保留最高严重性
- 相似问题归类合并
- 去重后生成唯一问题列表

### Phase 4: 生成最终报告

**报告结构**:
1. Summary（概览统计）
2. 各 Agent 详细报告
3. 问题汇总表
4. 最终决策

---

## 超时和错误处理

| 场景 | 处理策略 |
|------|----------|
| 单个 Agent 超时 | 标记为「未完成」，继续其他 Agent |
| 部分 Agent 失败 | 已完成的 Agent 报告仍采纳 |
| 全部 Agent 失败 | 返回错误，提示用户重试 |
| 文件读取失败 | 跳过该文件，在报告中标注 |

---

## 审查决策

### 决策类型

| 决策 | 条件 | 说明 |
|------|------|------|
| **Approve** | 无 blocking 问题 | 代码可以合并 |
| **Comment** | 有 important/nit 问题 | 建议修改但不阻止 |
| **Request Changes** | 有 blocking 问题 | 必须修改后才能合并 |

### 决策流程

```
检查 blocking 问题
    │
    ├── 有 → Request Changes
    │
    └── 无
            │
            ├── 有 important 问题 → Comment
            │
            └── 无/只有 nit/suggestion → Approve
```
