# 四阶段代码审查

## 阶段总览

| 阶段 | 名称 | 时间占比 | 核心目标 |
|------|------|----------|----------|
| Phase 1 | Context Gathering | 10% | 理解业务上下文 |
| Phase 2 | High-Level Review | 30% | 架构和设计评估 |
| Phase 3 | Line-by-Line Review | 50% | 逐行深度分析 |
| Phase 4 | Summary & Decision | 10% | 总结和决策 |

---

## Phase 1: Context Gathering (上下文收集)

### 目标
理解代码的业务背景和审查范围

### 任务清单
- [ ] 阅读 PR 描述或审查请求说明
- [ ] 了解业务需求和变更目的
- [ ] 确认涉及的文件和模块
- [ ] 检查 CI/CD 状态（测试通过？构建成功？）
- [ ] 识别技术栈和框架版本

### 输出
```markdown
### Context Summary
- **Purpose**: [业务目的]
- **Scope**: [审查范围]
- **Tech Stack**: [技术栈]
- **CI Status**: [CI状态]
```

---

## Phase 2: High-Level Review (高层审查)

### 目标
评估架构设计和整体质量

### 任务清单
- [ ] 评估模块划分是否合理
- [ ] 检查依赖关系是否清晰
- [ ] 评估抽象层次是否一致
- [ ] 检查错误处理策略
- [ ] 评估可测试性
- [ ] 检查是否有安全设计

### 关注点
- **架构模式**: MVC、Clean Architecture、微服务等
- **设计原则**: SOLID、DRY、KISS
- **依赖管理**: 循环依赖、过深调用栈
- **可维护性**: 代码组织、模块边界

---

## Phase 3: Line-by-Line Review (逐行审查)

### 目标
深度分析每一行代码

### 任务清单
- [ ] 逻辑正确性验证
- [ ] 边界条件检查
- [ ] 空指针/undefined 检查
- [ ] 资源泄漏检查
- [ ] 并发安全问题
- [ ] 安全漏洞扫描
- [ ] 性能热点识别
- [ ] 代码风格一致性

### 语言特定检查

#### TypeScript
- [ ] 类型安全（无 any）
- [ ] 泛型正确使用
- [ ] strict 模式合规
- [ ] async/await 正确使用
- [ ] 错误处理完整性

#### Python
- [ ] 类型注解完整性
- [ ] 异步编程规范
- [ ] 异常处理正确性
- [ ] 可变默认参数检查
- [ ] 装饰器使用规范

#### Go
- [ ] 错误处理规范
- [ ] Context 传播
- [ ] Goroutine 泄漏检查
- [ ] 切片/Map 并发安全
- [ ] 依赖管理规范

---

## Phase 4: Summary & Decision (总结决策)

### 目标
生成结构化报告并给出决策

### 任务清单
- [ ] 汇总所有发现的问题
- [ ] 按严重性分类
- [ ] 生成审查总结
- [ ] 给出最终决策

### 最终决策标准

| 条件 | 决策 |
|------|------|
| 存在 `🔴 [blocking]` | Request Changes |
| 存在 `🟡 [important]` 且无 blocking | Comment |
| 只有 `🟢 [nit]` 或 `💡 [suggestion]` | Approve |
| 代码质量优秀 | Approve with Praise |

---

## 时间管理

建议单个文件审查时间：
- **< 100 行**: 3-5 分钟
- **100-500 行**: 5-15 分钟
- **> 500 行**: 按需延长，最多 30 分钟

超过建议时间时：
1. 标记为「需要深入审查」
2. 优先处理 blocking 和 important 问题
3. 可选深入 nit 和 suggestion
